# Architecture

## Overview

Groot is a pnpm workspace monorepo with a clear separation between synced
boilerplate (`packages/`) and app-owned business code (`apps/web/`).

## Directory Structure

```
apps/web/                       # App-owned (NOT synced)
  src/client/                   # React frontend
    App.tsx                     # Route definitions
    main.tsx                    # React entry point
    app/                        # Business feature modules
  src/server/                   # Express backend
    index.ts                    # Server entry point
    routes.ts                   # Route registration
    app/                        # Business feature modules
  index.html                    # Vite HTML entry
  tsconfig.json                 # Client tsconfig
packages/                       # Boilerplate (always synced)
  ui/                           # @groot/ui - shadcn/ui design primitives
  client/                       # @groot/client - client infrastructure
  server/                       # @groot/server - server core + shared features
  database/                     # @groot/database - Prisma client wrapper
prisma/                         # Schema + migrations (app-owned)
tests/                          # Centralized unit tests
docs/                           # Documentation
.groot/                         # Sync tooling
```

## Package Dependency Graph

```
apps/web --> @groot/ui
apps/web --> @groot/client
apps/web --> @groot/server
apps/web --> @groot/database

@groot/client --> @groot/ui

@groot/server --> @groot/database
```

Packages never import from `apps/`. Apps import from packages.

## Package Details

### @groot/ui (`packages/ui`)

Pure shadcn/ui design system primitives (Button, Dialog, Input, Table, etc.).
Zero business logic. Components use `cn()` from `./lib/utils`.

**Exports:** `@groot/ui/button`, `@groot/ui/dialog`, etc.

### @groot/client (`packages/client`)

Client-side boilerplate infrastructure: Layout, apiClient, Zustand stores,
React hooks, utility functions, CodeMirror editor, passkey/settings services.

**Exports:** `@groot/client/lib/api`, `@groot/client/components/Layout`, etc.
Also exports the Tailwind entry CSS: `@groot/client/index.css`.

### @groot/server (`packages/server`)

Server-side infrastructure with two subpath export surfaces:

- `@groot/server/core/*` - logger, errors, job queue, KV store, AI client, middlewares, config, storage
- `@groot/server/shared/*` - reusable features: auth, passkey, jobs, settings, storage

### @groot/database (`packages/database`)

Prisma client singleton and generated client output.

**Exports:**

- `@groot/database` - prisma singleton, PrismaClient type, Prisma namespace
- `@groot/database/client` - raw prisma instance
- `@groot/database/types` - type-only re-exports

The generator output path (`packages/database/generated/prisma`) is gitignored
and regenerated by `pnpm prisma generate`.

## Synced vs App-Owned Surface

| Surface                                          | Synced? | Examples             |
| ------------------------------------------------ | ------- | -------------------- |
| `packages/**`                                    | Yes     | All boilerplate code |
| `tests/server/core/**`, `tests/server/shared/**` | Yes     | Infra tests          |
| `tests/client/components/ui/**`                  | Yes     | UI component tests   |
| `scripts/**`, `.groot/**`                        | Yes     | Tooling              |
| `docs/**`                                        | Yes     | Documentation        |
| `*.config.*`, `tsconfig.json`                    | Yes     | Build/config files   |
| `apps/web/src/**`                                | **No**  | Business code        |
| `prisma/schema.prisma`                           | **No**  | App schema           |
| `prisma/migrations/**`                           | **No**  | App migrations       |

## Schema Contract

Downstream `schema.prisma` files must keep these models (the boilerplate's
shared features depend on them):

- `User` - used by `@groot/server/shared/auth`
- `Passkey` - used by `@groot/server/shared/passkey`
- `JobLog` - used by `@groot/server/core/logger`
- `Keyv` (@@ignore) - used by `@groot/server/core/kv`

Apps can freely add additional models.

## Adding a Business Feature Downstream

1. **Server module** (`apps/web/src/server/app/<feature>/`):
   - `<feature>.routes.ts` - route definitions
   - `<feature>.controller.ts` - request handlers
   - `<feature>.service.ts` - business logic
   - `<feature>.validation.ts` - Zod schemas
   - `<feature>.model.ts` - Prisma queries
   - `<feature>.jobs.ts` - background job handlers (optional)
   - `index.ts` - exports

2. **Register routes** in `apps/web/src/server/routes.ts`

3. **Client module** (`apps/web/src/client/app/<feature>/`):
   - `pages/` - page components
   - `hooks/` - React Query hooks
   - `components/` - feature-specific components (optional)

4. **Register routes** in `apps/web/src/client/App.tsx`

## Patch and Upstream Workflow

1. Edit a synced file locally (e.g. fix a bug in `packages/server/src/core/...`)
2. Test your fix in the downstream app
3. Run `pnpm groot:upstream` to push the change back to groot
4. The fix flows to all other downstream repos on their next `pnpm groot:sync`
